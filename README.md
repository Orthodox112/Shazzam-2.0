# EE200 &middot; Audio Fingerprinting (Q3A / Q3B)

A small Shazam-style audio identifier built for the EE200 Signals, Systems
& Networks course project. Indexes a 50-song library as spectrogram
fingerprints, then identifies short query clips against it — with a
Streamlit app that shows every intermediate step (spectrogram,
constellation of peaks, offset histogram) exactly as required by Q3B.

## 2. Local setup

```bash
# 1. Create and activate a virtual environment
python -m venv .venv
source .venv/bin/activate        # Windows: .venv\Scripts\activate

# 2. Install dependencies
pip install -r requirements.txt

# 3. Drop your 50 song files into data/songs/
#    (use the exact filenames provided — the filename without extension
#     becomes the label your identifier outputs, per the project spec)
cp /path/to/your/50/songs/* data/songs/

# 4. Build the fingerprint database (run once, ~1-3 min for 50 short songs)
python build_database.py

# 5. (optional) generate a few "Try a sample" clips for the Identify tab
python make_samples.py --n 5 --seconds 10

# 6. Run the app
streamlit run app.py
```

The app opens at `http://localhost:8501` with three tabs: **Library**,
**Identify**, **Batch**.

## 3. How the pipeline works (maps directly to Q3A's sub-questions)

| Stage | File | What it does |
|---|---|---|
| Spectrogram | `spectrogram.py` | STFT (`scipy.signal.stft`) of the clip -> magnitude in dB, normalized to 0 dB peak. Time/frequency resolution trade-off is controlled by `N_FFT` / `HOP_LENGTH` in `config.py`. |
| Constellation | `peaks.py` | 2-D local-maximum filter picks out the loudest, sparse set of (time, freq) points — the points that survive EQ, loudness changes, and mild noise. |
| Hashing | `hashing.py` | Each peak ("anchor") is paired with up to `FAN_OUT` nearby later peaks ("partners") within a target time window, producing hashes `(f1, f2, Δt)`. A "single peaks on their own" baseline is also provided for the Q3A comparison. |
| Database | `database.py` | Builds an inverted index `hash -> [(song_id, time), ...]` for every song, persisted as JSON. Two parallel indices are kept — one for paired hashes, one for single-peak hashes — so the comparison experiment can be run live in the app. |
| Matching | `matcher.py` | For every query hash, looks up matching DB postings and votes for `(song, db_time - query_time)`. A true match produces one dominant spike in the offset histogram; wrong songs spread votes thinly. The top candidate's vote count is the **cluster score**; we also report how many times bigger it is than the runner-up. |
| Robustness | `robustness.py` | Helpers to add white noise at a target SNR, pitch-shift, and time-stretch a query, used for the report's robustness experiments. |

All tunable numbers (FFT size, hop length, peak density, fan-out, match
thresholds) live in **`fingerprint/config.py`** — tune there if your 50
songs need different settings (e.g. very short or very dense tracks).

## 4. The three tabs

### Library
Read-only grid of every indexed song and how many fingerprint hashes it
contributed. Indexing itself is admin-only (`build_database.py`) — the
spec treats the database as something built once and shipped with the
app, not something end users rebuild from the UI.

### Identify (single-clip mode)
Upload a clip (or click **Try** on a bundled sample) and click
**Identify**. Displays, top to bottom:
1. **Pipeline timing chips** — Spectrogram / Constellation / Hashing /
   DB Lookup / Scoring, each with its elapsed time and a one-line stat
   (FFT shape, peak count, hash count, offset).
2. **Match found / no match panel** — winning song, cluster score, and
   how many times bigger that score is than the runner-up.
3. **Candidate scores** — top-5 songs ranked by vote count.
4. **Step 1: spectrogram → constellation** — the two plots side by side.
5. **Step 2: offset histogram** — votes per offset for the winning
   candidate, with the winning offset highlighted.

An **Advanced** expander lets you re-run the same clip using single-peak
hashes instead of paired hashes, to reproduce the Q3A comparison live.

### Batch
Upload many query clips at once, click **Run batch**, and download
`results.csv` with exactly the columns `filename, prediction` (prediction
is `"none"` when no candidate clears the confidence threshold) — the
exact automated-grading format required by the spec.

## 5. Tuning match confidence

Two knobs in `config.py` control when the matcher reports "no match":

```python
MIN_VOTES_FOR_MATCH = 5      # cluster score must be at least this many votes
MIN_RUNNER_UP_RATIO = 1.5    # AND beat the 2nd place candidate by this factor
```

If you find real matches are being rejected, lower these; if wrong songs
are being confidently (mis)matched, raise them. Re-run
`python tests/test_pipeline.py` after any change to confirm nothing broke.

## 6. Running the experiments for the report (Q3A)

```python
from fingerprint.audio_io import load_audio
from fingerprint.database import FingerprintDB
from fingerprint.matcher import match_clip
from fingerprint.robustness import run_noise_sweep, run_pitch_sweep, run_stretch_sweep

db = FingerprintDB.load()
y, sr = load_audio("data/songs/Hey Jude.mp3")
query = y[int(20*sr): int(30*sr)]   # a 10s clip

noise_results  = run_noise_sweep(query, sr, db, match_clip)
pitch_results  = run_pitch_sweep(query, sr, db, match_clip)
stretch_results = run_stretch_sweep(query, sr, db, match_clip)
```

Each returns a list of dicts (`predicted`, `score`, `runner_up_ratio`,
`is_match`) you can drop straight into a pandas DataFrame and plot for
the report's robustness section.

## 7. Deploying to Streamlit Community Cloud

1. Push this whole folder to a **public** GitHub repo (the indexed
   `data/db/fingerprints.json` must be committed so the deployed app
   works immediately — see the Dataset Usage / Song Database note in
   the project spec).
2. If your 50 song files are small enough for GitHub (check repo size
   limits), commit `data/songs/` too so graders can re-index if needed;
   otherwise it's enough to ship just the pre-built `fingerprints.json`.
3. Go to [share.streamlit.io](https://share.streamlit.io), sign in with
   GitHub, click **New app**, pick this repo/branch, and set the main
   file path to `app.py`.
4. Deploy. First boot installs `requirements.txt` (librosa's first
   import can take ~30s to compile its cache — subsequent loads are fast).
5. Test both modes on the live URL before submitting:
   - **Identify** tab: upload one clip, confirm spectrogram/constellation/
     histogram all render and the right song is found.
   - **Batch** tab: upload several clips, confirm `results.csv` downloads
     with the exact `filename,prediction` header.
6. Submit the live app link **and** the GitHub source link in your report PDF.
